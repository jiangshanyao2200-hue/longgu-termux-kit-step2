# 工具状态（图灵化约定）

本项目的 MCP 工具输出由两部分组成：

- `user_message`：给用户看的简短说明（TUI 里展示）。
- `log_lines`：给智能体看的结构化日志（也会在 UI 中摘要展示）。

## 1) “失败”的唯一判定

**只有当 `log_lines` 中出现 `状态:fail` 时，才视为工具执行失败。**

其他任何 `状态:*`（例如 `状态:0`、`状态:ok_timeout`、`状态:ok_not_found`、`状态:timeout`）
都属于“情况汇报”，UI 不应渲染为“执行失败：...”。

## 2) 状态行格式

第一条日志建议固定为状态行：

- `状态:0`：成功（或无错误的正常完成）。
- `状态:fail`：失败（需要用户/智能体修复或重试）。
- `状态:ok_*`：成功但有重要条件/退化/提示（例如：跨文件系统退化为 copy-only）。
- `状态:timeout / 状态:ok_timeout`：超时（不等价于失败，是否重试取决于工具副作用）。

状态行之后可以追加补充字段（保持可读即可），如：

- `saved:...`：输出/导出落盘路径
- `log:...`：对应日志路径
- `meta:...`：关键元信息（行数、大小、job_id 等）

## 3) 工具实现侧要求

- 所有工具**应显式写入**一条 `状态:*` 到 `log_lines`（避免依赖兜底推断）。
- “预期的非 0”应通过 `ok_exit_codes` 之类机制声明为“情况汇报”，而不是抛异常。
- “副作用工具”（写文件/删文件/启动 PTY/杀进程）**超时后不得自动重试**，必须返回可解释状态与可恢复信息（如回收站 id/备份路径）。
