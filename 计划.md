bash：增强版支持pty交互 和跑长任务 确保能实现人机协同 且可以支持开多个pty窗口同时追踪，我们就需要确保返回的逻辑。AI跑长任务时可以同时追踪多个pty的实现，比如同时install多个依赖 程序，需要完成后自动退出pty并将执行结果储存，等到一起完成后才打包返回结果。但当AI执行时会立刻返回结果，返回基本信息各个pty是否成功运行。多个pty执行结果
那么 bash的模式拍板为
普通模式，支持多命令
交互模式，通过pty启动命令，让用户可以操作。操作期间log保持更新，5分钟发送一遍日志情况给AI追踪，追踪期间用户可以和AI聊天询问进度，用户也可以alt+↑ 将快照存入，输入消息内容时发送时会一并发送。为了避免系统发送日志给AI追踪进度和用户聊天产生并发。这些消息都纳入消息队列管理器（我们已有），并保证用户输入优先级最低。这是处于安全考虑。因为如果有问题用户可以esc退出，通常发消息那就是不太重要。
多路并发模式，允许AI同时跑多个命令，并以pty形式呈现。这些pty窗口将重叠展示出去（需要考虑多pty重叠时运用同一个UI逻辑，更像是一个ptyUI 但多个标签页切换） 用户可以输入pgup和pgdn快速切换不同的pty窗口 也就是说只要pty界面存在并且未隐藏，那么pgup pgdn就不再作用于消息区域的聊天选择，而是作用于pty的多标签页焦点切换。同时也会把输入框作为一个焦点一并加入切换，这样用户实际上在切换整个焦点，也就是pty出现时焦点是在 主页层？那么就是输入框可以输入，如果是在pty层，那么就是pty层可以输入，如果是pty标签2层，那就是pty标签3可以输入。当用户使用home 隐藏 这个是隐藏所有pty，此时焦点自动回到主页，并且pgup和pgdn作用于主页，可以选择消息。也就是说是判断pty是否呈现来决定pgup dn按键的运用。
关于消息返回，执行多个pty任务时，工具首先返回执行结果。然后为不同的pty建立不同的log，返回log路径和执行结果给AI。
通常为了避免并发，和多次发送导致token消耗，所有pty需要跑完后打包一起传递执行结果。由于工具一开始就返回了执行结果。所以结尾是属于工具主动消息的程序传递结果。大概意思，AI那里工具轮询已断，但工具本身未断，工具执行完毕仍需返回结果。该逻辑运用于交互模式与多pty模式。不同的是 交互模式返回一个最终结果。多pty，返回多个log和多个结果。
我们之前的返回结果应该是，如果阈值内直接返回，阈值外截断在返回，并导出到cache。但实际情况我们现在已经有了log，这是启动时就建立的。所以省略了导出这一步。无论是否超出阈值都会导出，那我们就统一回传上下省略中间关键信息，外加log路径和执行结果返回给AI。阈值内的直接返回全量，阈值外的截断。中间，确保返回的log内容限定在一定的阈值内。通常AI只需要看到执行结果。如果需要看执行详细内容，可以阅读log。需要注意返回要附带对应log的行数，大小，方便AI对该log有一个基本的认知理解。这个无论是否触发阶段，都返回对应log信息，因为它始终有log文件在。路径文件名行数大小等

adb shell：通常没有需要增强的，可以后续再考虑是否继续增强，就是超出截断导出的log，需要增加log的路径文件名行数，大小，等让AI直观理解对应log文件。无log，没触发导出不显示。

termux api：同上，功能不多，暂不考虑增强。只对导出的log做一个返回关键信息 路径文件名，行数大小信息告知，无额外导出log不告知。

首先是Search 我们当前是整合的，但我们需要分开了
先是file manager的完善与重构
file manager是针对AI所做的一个文件管理器
它的核心目标是方便AI进行批量操作文件内容
File Search，支持三种模式，文件检索，同时检索文件夹名，文件名。并分别直接回传 关键信息，如文件的大小等AI可能会需要的信息。并按路径整理回传路径，避免每个文件都显示路径造成多余token消耗，大概这个检索会像文件树一样根据路径将检索结果列出来。并且智能化，设定匹配项阈值，在一个可控范围内回传信息，如果多了，还是导出到log 下fm search cache 并返回一部分内容。我估计阈值在500个结果内可接受，但你要评估给出一个合理值。超出后导出log，并返回log的路径文件名，行数，文件大小等
只检索文件夹和只检索文件名。为了避免AI错误的在整个大目录下进行检索，对该工具进行时间限制，我还没有好的预案。这个需要仔细考究一下

然后是creat，模式1 支持creat单个文件夹，输入路径（在该路径下creat文件夹），输入文件夹名称。 支持creat单个文件。输入路径，文件名，文件类型，单个文件支持写入基本内容。模式2 支持同目录creat多个文件。是需要输入一个路径，需要依次输入文件名文件类型。模式3 在不同路径创建多个文件。模式4 直接创建工程结构，多个文件夹，多个文件。关于是否支持创建基本内容，这个需要仔细聊聊，多个文件如果同时创建内容，可能导致模型一次性输出大量token且容易是无效的。我们提供全部能力，但不意味着模型每次都要用。

然后是edit，支持重命名文件，文件夹，支持重命名多个文件夹，文件名，文件类型等。支持移动文件，文件夹到某个路径。支持批量移动到某个路径。支持批量移动多个文件到不同路径。同样复制也是。由于这些操作无法确认，我们不支持对重复的文件进行覆盖。如果有同名情况下，需要先将同名删除才能移动或复制过去。edit的功能大致如此 它管理文件夹和文件，如果需要更多功能，可以聊聊。

然后是delete，支持delete单个或多个文件夹，或者整个目录（不强制，如果正在使用无法删除）我们的删除是在system建立回收站来安全删除。这个删除操作会把要删除的文件移动到回收站（我们已有trash）如果有更多功能或者安全考虑，可以仔细讨论

stats：可以获取文件详细信息，尽量是AI所需的关键信息。支持同时获取多个文件的信息。可以最多获取指定路径下的全部文件的信息，包括该目录下的文件夹（不深入子文件夹）文件，获取文件名 大小 行数（只针对文本文件md txt等或者py等常见的文本编辑器支持的文件）如html。上限阈值宽松

list 支持列目录，支持列出结构树 结构树包含文件大小 上次修改手时间等关键信息，结构树子目录做个层上限。避免直接把整个手机目录全列出来这种荒诞行为。 并做一个返回安全策略。

然后是editor，editor通常是编辑文本写代码的重中之重。所以我们决定为之建立一个更强大的编辑器来提供支持。

首先是write file 我们已有，但当前write file同事作用于manager，需要分离。确保writefile只支持覆写和写入，不再支持创建。如果AI直接写，但文件不存在，我们需要工具将AI写入的暂存为log，（log目录下 ed write cache，告诉AI写入已经暂存，因为文件不存在，所以无法写入，写入内容已临时保存到路径，文件大小 ，文件行数，确保我们如果返回文件信息，都尽量的返回关键信息。除了seach不返回行数信息。）避免AI需要重写，AI则只需要配合manager，rename该log并将其移动或复制到它原本想写入的路径。我们当前write已经趋于完善 还支持原子写入，需要聊的是关于write覆写，当AI标记明确覆写参数那就直接覆写，但要讲源文件移动到trash，避免丢失。
